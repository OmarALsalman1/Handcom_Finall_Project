# Handcom Backend

A smart maintenance services platform that connects **Service Users** with
**Service Providers** (plumbers, electricians, painters, carpenters, etc.)
through an AI-powered Django REST Framework backend.

The integrated AI assistant analyses a problem via text, image, or voice and
either provides a direct DIY solution (for simple issues) or recommends the
best-matched providers based on rating, location, and availability.

## Authentication

Both roles receive a JWT with a `role` claim:

```
POST /api/v1/auth/service-user/login/     в†’ { access, refresh, role: "service_user" }
POST /api/v1/auth/service-provider/login/ в†’ { access, refresh, role: "service_provider" }
```

Include the access token as `Authorization: Bearer <token>` on every protected request.

## Architecture Notes

### AI provider вЂ” pluggable via env var

`AI_PROVIDER=rule_based` (default) uses keyword matching; no external API needed.  
`AI_PROVIDER=gemini` with `GEMINI_API_KEY` set uses Gemini 2.0 Flash and falls
back to rule-based on any API error.

See `apps/ai_assistant/services.py` в†’ `AIProvider` ABC and `get_ai_provider()`.

### Realtime chat вЂ” stubbed, ready to swap

After every message save, `RealtimeProvider.publish()` is called.
The default implementation is a no-op. Replace with Stream Chat or Django Channels
by subclassing `RealtimeProvider` in `apps/conversations/services.py`.

### Media storage вЂ” local в†’ Cloudinary swap

`ImageField` / `FileField` write to `MEDIA_ROOT` in dev.
Swap to Cloudinary by replacing the storage backend behind a `MediaStorage` interface.

### Status machine вЂ” enforced in the service layer

All `ServiceRequest` transitions are validated in `apps/services/services.py`,
never in views or serializers. State: `pending в†’ accepted в†’ in_progress в†’ completed`;
`cancelled` allowed from `pending` or `accepted`.

### JWT dual-role auth

`HandcomJWTAuthentication` (extends `JWTAuthentication`) reads the `role` claim:
- `service_user` в†’ resolves via Django's standard user lookup
- `service_provider` в†’ resolves via `ServiceProvider` table using `provider_id` claim
